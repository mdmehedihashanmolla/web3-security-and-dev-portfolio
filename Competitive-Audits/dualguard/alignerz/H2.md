# [H-02] Unbounded loops in A26ZDividendDistributor permanently break dividend distribution at scale

Severity: High
Category: Gas / Denial of Service (DoS)

Platform: DualGuard
Protocol: AlignerZ
Ecosystem: EVM
Language: Solidity

---

## Summary

The `A26ZDividendDistributor` relies on an unbounded loop to calculate and distribute dividends across all minted NFTs. Because the total number of NFTs strictly increases over time, the gas required to execute this loop grows linearly. As the protocol achieves organic adoption, the transaction gas cost will inevitably exceed the network's block gas limit. This creates a permanent, unrecoverable Denial of Service (DoS) that traps all stablecoin dividends within the contract and completely breaks the reward mechanism.

---

## Vulnerability Details

The protocol's dividend epoch is triggered when the owner calls `setUpTheDividends()`. This function executes `_setAmounts()` and `_setDividends()`. Both internal functions iterate from `0` to `nft.getTotalMinted()`.

While unbounded loops are a known anti-pattern, this specific implementation is exceptionally dangerous due to the heavy EVM operations performed during *each* iteration.

```solidity
function getTotalUnclaimedAmounts() public returns (uint256 _totalUnclaimedAmounts) {
    uint256 len = nft.getTotalMinted();
    for (uint256 i; i < len;) {
        (, bool isOwned) = safeOwnerOf(i); // @audit - 1st External call
        if (isOwned) _totalUnclaimedAmounts += getUnclaimedAmounts(i);
        unchecked { ++i; }
    }
}

function getUnclaimedAmounts(uint256 nftId) public returns (uint256 amount) {
    // @audit - 2nd External call retrieving a massive struct
    if (address(token) == address(vesting.allocationOf(nftId).token)) return 0;
    // ...
```

Inside the loop, the contract queries `vesting.allocationOf(nftId)`. The returned `Allocation` struct is not a simple data type; it contains four dynamic arrays: `amounts`, `vestingPeriods`, `vestingStartTimes`, and `claimedSeconds`.

When a contract retrieves dynamic arrays via an external call, the EVM must perform a `RETURNDATACOPY` and allocate dynamic memory to store the incoming struct. Because Solidity does not free memory between loop iterations, the memory footprint continuously expands.

As memory expansion gas costs grow quadratically and external calls incur significant base costs, this loop will rapidly exhaust the block gas limit (e.g., 30M gas on Ethereum). Because the loop cannot be paused, chunked, or bypassed, the admin will be permanently locked out of calling `setUpTheDividends()`.

---

## Root Cause

The contract architecture uses unbounded iteration over a strictly growing dataset (`nft.getTotalMinted()`) to execute heavy cross-contract calls and dynamic memory allocations, lacking any pagination or batching functionality.

---

## Impact

**High.** This is a guaranteed architectural failure (Self-DoS). It does not rely on a malicious actor; standard protocol growth will naturally brick the contract. Once the threshold of minted NFTs is reached, the trusted admin is rendered powerless. All stablecoins meant for TVS holders will become permanently locked in the distributor contract, resulting in a total failure of the core economic reward system.

---

## Failure Scenario

1. The protocol launches successfully. During the first few weeks, the NFT supply remains low (e.g., 50-100 NFTs). The admin successfully calls `setUpTheDividends()`.
2. The protocol gains widespread adoption, and users mint over 2,000 NFTs.
3. At the end of the next epoch, the admin attempts to call `setUpTheDividends()`.
4. The contract attempts to loop through all 2,000+ NFTs, triggering over 4,000 external calls and thousands of dynamic memory allocations in a single transaction.
5. The gas cost exceeds the network's block gas limit. The transaction suffers an Out-of-Gas (OOG) revert.
6. The admin cannot split the transaction. The dividend distribution process is permanently halted, and user rewards are locked forever.

---

## Recommendation

Do not iterate over the entire token supply in a single transaction. Refactor the architecture to support paginated, batched processing using `startIndex` and `batchSize` parameters.

This requires splitting the setup process so the admin can process the arrays in manageable chunks across multiple transactions:

```solidity
// Track the current index in state
uint256 public nextTotalIndex;

// Allow the admin to process a specific number of NFTs per transaction
function processUnclaimedAmountsBatch(uint256 batchSize) external onlyOwner {
    uint256 len = nft.getTotalMinted();
    uint256 end = nextTotalIndex + batchSize;
    if (end > len) end = len;

    for (uint256 i = nextTotalIndex; i < end;) {
        (, bool isOwned) = safeOwnerOf(i);
        if (isOwned) totalUnclaimedAmounts += getUnclaimedAmounts(i);
        unchecked { ++i; }
    }
    
    nextTotalIndex = end;
}
```

Apply the exact same batching pattern to `_setDividends()`. This guarantees that no matter how large the NFT supply becomes, the protocol can always safely distribute dividends without exceeding block gas limits.

---

## Conclusion

The current dividend distribution logic relies on an $O(N)$ gas-intensive loop that scales linearly with the total NFT supply. By retrieving complex dynamic structs across contract boundaries inside this loop, the contract will inevitably suffer an Out-of-Gas revert at scale, permanently locking protocol rewards. Implementing a paginated state-update pattern ensures the protocol remains resilient and operational regardless of user adoption levels.
