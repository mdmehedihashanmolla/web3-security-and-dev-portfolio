# [H-01] Cumulative fee calculation permanently locks user funds during multi-flow TVS operations

Severity: High
Category: Accounting / Logic

Platform: DualGuard
Protocol: AlignerZ
Ecosystem: EVM
Language: Solidity

---

## Summary

When users split or merge Token Vesting Schedules (TVS) with multiple token flows, they are overcharged. The protocol incorrectly subtracts a cumulative running total of fees from each individual flow rather than a per-flow fee. The excess deducted tokens are neither sent to the treasury nor returned to the user, leaving them permanently locked in the `AlignerzVesting` contract.

---

## Vulnerability Details

The vulnerability exists in `calculateFeeAndNewAmountForOneTVS`, which handles fee calculations for permissionless TVS splits and merges.

When processing operations on a TVS with multiple flows, the function loops through the `amounts` array:

```solidity
function calculateFeeAndNewAmountForOneTVS(uint256 feeRate, uint256[] memory amounts, uint256 length)
    public
    pure
    returns (uint256 feeAmount, uint256[] memory newAmounts)
{
    for (uint256 i; i < length;) {
        feeAmount += calculateFeeAmount(feeRate, amounts[i]);
        newAmounts[i] = amounts[i] - feeAmount; // @audit - Subtracts global accumulator from local flow
    }
}
```

`feeAmount` correctly tracks the total fee owed to the treasury. However, `newAmounts[i]` subtracts the *entire accumulated fee up to that iteration* rather than just the fee for that specific flow.

When the transaction completes, the treasury receives the correct total fee, but the user is severely under-allocated. Because this missing delta is never accounted for in the state transition, the tokens are permanently orphaned in the contract, breaking the 1:1 asset backing invariant.

---

## Root Cause

The function subtracts a global accumulator (`feeAmount`) from a local variable (`amounts[i]`) instead of calculating and subtracting a discrete, per-flow fee.

---

## Impact

**High.** Users suffer a direct, irrecoverable loss of tokens on any multi-flow TVS split or merge. The magnitude of the loss scales with the number of flows in the vesting schedule.

---

## Loss Scenario

1. A user splits a TVS containing 3 flows of 10,000 tokens each. The fee rate is 1% (100 tokens per flow).
2. The internal loop processes the flows:
   * **Index 0:** `feeAmount` = 100. `newAmount[0]` = 10,000 - 100 = 9,900. *(Correct)*
   * **Index 1:** `feeAmount` = 200. `newAmount[1]` = 10,000 - 200 = 9,800. *(User loses 100)*
   * **Index 2:** `feeAmount` = 300. `newAmount[2]` = 10,000 - 300 = 9,700. *(User loses 200)*
3. The treasury is sent exactly 300 tokens.
4. The user is allocated 29,400 tokens across their new flows.
5. The remaining 300 tokens (30,000 original - 300 fee - 29,400 allocated) remain stuck in the contract forever.

---

## Proof of Concept

This Foundry test isolates `FeesManager.sol` to prove the exact mathematical loss caused by the loop.

Create `FeesManager.t.sol` in your test directory and run: `forge test --mt test_TVS_Loss_CumulativeFee -vv`

```solidity
// SPDX-License-Identifier: MIT
pragma solidity =0.8.29;

import "forge-std/Test.sol";
import "../src/contracts/vesting/feesManager/FeesManager.sol";

contract FeesManagerHarness is FeesManager {
    function initialize() external initializer {
        __FeesManager_init();
    }
}

contract FeesManager_PoC is Test {
    FeesManagerHarness public feesManager;

    function setUp() public {
        feesManager = new FeesManagerHarness();
        feesManager.initialize();
    }

    function test_TVS_Loss_CumulativeFee() public {
        uint256 feeRate = 100; // 1% fee 
        uint256 length = 3;
        
        uint256[] memory amounts = new uint256[](length);
        amounts[0] = 10_000 ether;
        amounts[1] = 10_000 ether;
        amounts[2] = 10_000 ether;

        // Simulate the buggy calculation 
        uint256 feeAmount;
        uint256[] memory newAmounts = new uint256[](length);
        for (uint256 i; i < length; i++) {
            feeAmount += feesManager.calculateFeeAmount(feeRate, amounts[i]);
            newAmounts[i] = amounts[i] - feeAmount; 
        }

        uint256 totalNew = newAmounts[0] + newAmounts[1] + newAmounts[2]; 
        
        // Treasury gets the correct 300 ether
        assertEq(feeAmount, 300 ether, "Treasury fee is correct");

        // User should get 29,700 ether. They get 29,400.
        // 300 ether is permanently lost to the contract.
        assertEq(totalNew, 29_400 ether, "User was under-allocated");
    }
}
```

---

## Recommendation

Isolate the per-flow fee into a local variable during the loop. Subtract this local fee from the individual flow amount, then add it to the global `feeAmount` accumulator.


```solidity
function calculateFeeAndNewAmountForOneTVS(uint256 feeRate, uint256[] memory amounts, uint256 length)
    public
    pure
    returns (uint256 feeAmount, uint256[] memory newAmounts)
{
    newAmounts = new uint256[](length); 
    
    for (uint256 i; i < length;) {
        uint256 currentFlowFee = calculateFeeAmount(feeRate, amounts[i]);
        
        feeAmount += currentFlowFee;
        newAmounts[i] = amounts[i] - currentFlowFee;
        
        unchecked { ++i; }
    }
}
```
